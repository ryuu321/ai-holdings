ChatGPT Prompts for Technical Writers

15 free prompts to write clearer documentation, collaborate better with engineers, and move faster on every type of technical content.

Technical writing is one of the most underestimated skills in software — and one of the most time-intensive. Great docs reduce support tickets, accelerate onboarding, and make engineers look smarter. These 15 prompts help you write faster, plan better, and communicate more effectively with every stakeholder who touches your documentation.

1. Documentation Writing

Prompt 1 — Getting Started Guide

Write a Getting Started guide outline and first section for a technical product. Product: [describe what it does — keep it general, no proprietary detail] Target reader: [developer / non-technical end user / system administrator / data analyst] Their starting point: [what they know before reading this / what environment they are in] Goal of the guide: [get to "Hello World" in under 10 minutes / connect the first integration / create their first project] What typically causes confusion in the first 30 minutes: [list 3 common friction points] Create: (1) Getting Started guide outline (sections and what each covers), (2) Prerequisite section (what they need before starting — environment, credentials, dependencies), (3) Step 1 in full — with exactly the level of detail needed (not too much, not too little), (4) How to write error-state guidance (when something goes wrong at step 1, what does the reader see and what should they do), (5) The "next steps" section that keeps them moving after they succeed. Good Getting Started guides are ruthlessly focused — one path, one success state.
Pro tip: Write your Getting Started guide for someone who is smart but knows nothing about your product. Then have someone who actually fits that description try to follow it. You will discover every unclear sentence.

Prompt 2 — API Reference Page

Help me write an API reference page for a specific endpoint. Endpoint: [HTTP method + path — e.g., POST /users/{id}/messages — use a generic example] What it does: [describe in plain language] Request parameters: [path params / query params / request body fields — describe the structure] Response: [what a successful response contains / what error responses look like] Authentication: [how the endpoint is authenticated] Common use cases: [what developers are typically trying to do when they call this] Write: (1) Endpoint description (2–3 sentences — what it does and when to use it), (2) Parameters table format (name / type / required or optional / description), (3) Example request (realistic but uses placeholder values — no real credentials), (4) Example response (realistic structure), (5) Error codes relevant to this endpoint, (6) Code sample snippet structure (how to format a code example without the actual code). Also: the most common mistake developers make with this type of endpoint — and how to address it in the docs.

Prompt 3 — Conceptual Documentation

Write conceptual documentation explaining a technical concept or architecture. Concept: [authentication flows / webhook architecture / data models / rate limiting / caching / permission systems / event-driven architecture — describe the specific concept] Audience: [developers new to this concept / experienced developers new to this product / mixed] What they need to understand before they can use the product effectively: [the mental model they need] Write: (1) Introduction section (why this concept matters — start with the problem it solves, not the definition), (2) How it works (the mental model — use an analogy first, then the technical reality), (3) Key terms defined in plain language (3–5 terms they will encounter), (4) Common patterns or use cases (how this concept is typically applied), (5) What to read next (which tutorial or reference to tackle after understanding this concept). Conceptual docs that start with "why" instead of "what" get read; ones that start with definitions get skipped.

2. Planning & Structure

Prompt 4 — Documentation Audit

Help me audit my existing documentation and identify gaps. Documentation type: [developer docs / user guide / admin guide / API reference / help center] Current state: [describe what exists — sections, formats, rough age of content] Known problems: [outdated content / missing topics / poor discoverability / no clear audience / inconsistent style / gaps between what product does and what docs cover] Audience feedback I have received: [what users say is confusing or missing] Audit framework: (1) Dimensions to evaluate (coverage / accuracy / clarity / structure / discoverability / audience alignment), (2) For each dimension: what "good" looks like vs. red flags, (3) Content inventory template (how to catalog what exists and assess each page), (4) Prioritization framework (how to rank which gaps to fix first — by user impact × content effort), (5) How to communicate audit findings to engineering and product stakeholders who control prioritization.

Prompt 5 — Documentation Information Architecture

Help me design the information architecture for a documentation site. Product type: [developer tool / SaaS application / enterprise software / API / platform] Primary audiences: [list 2–4 — e.g., developers / administrators / end users / decision makers] Content types I have or need: [getting started / tutorials / how-to guides / API reference / conceptual guides / troubleshooting / release notes / FAQ] Navigation goals: [users should be able to find X in under 3 clicks / new users should be funneled to getting started / experienced users should reach reference quickly] Design: (1) Top-level navigation structure with rationale for each section, (2) Content type taxonomy (what type of content goes where and why), (3) How to handle content that serves multiple audiences, (4) Recommended URL structure, (5) The Diátaxis framework applied to this documentation set (if relevant — tutorials vs. how-to guides vs. reference vs. explanation), (6) How to onboard new writers to this architecture so they add content in the right place. Good IA is invisible — users find what they need without knowing why.
Pro tip: The Diátaxis framework (tutorials / how-to / reference / explanation) is the most useful IA model for technical documentation. Each type serves a different user need — mixing them in the same page confuses both readers and writers.

3. Collaboration

Prompt 6 — Engineering Collaboration Email

Write emails to collaborate more effectively with engineers on documentation. Scenario: [requesting a technical review / following up on unanswered questions / pushing back on unclear explanations from an SME / asking for a demo before documenting a feature / explaining why the docs need to be done before launch] Write templates for: (1) Requesting a technical review (what I need, why it matters, how long it will take, specific questions to answer), (2) Following up on a review that has stalled for 2+ weeks (warm but direct), (3) Getting an engineer to explain something they explained in a way I could not document, (4) Pushing back on "just document what the code does" (explaining why user-facing docs need context that the code does not provide), (5) The pre-launch documentation conversation (what needs to be done, who needs to do what, why the docs need to be ready before ship date not after). Technical writers who communicate like project owners get better results than those who wait to be given information.

Prompt 7 — Style Guide Development

Help me build a documentation style guide for my team or project. Documentation type: [developer docs / help center / API reference / internal docs] Team size: [solo writer / 2–5 writers / larger team with multiple contributors including engineers] Biggest consistency problems now: [voice and tone / formatting / terminology / code sample style / how to address the reader / British vs. American English — describe what is inconsistent] Tools used: [Markdown / AsciiDoc / MadCap Flare / Confluence / Google Docs / other] Style guide sections: (1) Voice and tone (what the docs sound like — 3 principles with examples), (2) Writing for the reader (you vs. they / active vs. passive / sentence length guidance), (3) Terminology decisions (must-use terms, do-not-use terms, preferred alternatives), (4) Formatting standards (headings, lists, code blocks, screenshots, links, notes and warnings), (5) Code sample standards (what to include, how to format, whether to use real values), (6) How to enforce it without being the style police. The best style guides make decisions once so writers do not have to make them again.

4. Specialized Content

Prompt 8 — Release Notes

Write release notes for a product update. Release type: [major version / minor release / patch / hotfix / API change] What changed: [describe features, fixes, and changes in general terms — do not include proprietary details] Audience: [developers / end users / admins / all] Tone: [technical and precise / accessible to non-developers / enterprise-formal] Write: (1) Release notes structure (how to organize by category — new features / improvements / bug fixes / deprecations / breaking changes), (2) Opening summary sentence (what this release is about in one line), (3) How to write a feature entry (name → what it does → why it matters → how to use it → link to full docs), (4) How to write a breaking change warning (clear, specific, what they need to do before upgrading, migration path), (5) What to include in the "coming next" section (if relevant). Release notes that help users decide whether to upgrade immediately vs. plan are more useful than lists of features.

Prompt 9 — Troubleshooting Guide

Write a troubleshooting guide for common errors or problems. Product area: [authentication / installation / API errors / integrations / performance / data sync — describe the area] Top 5 issues users encounter: [describe each in general terms] What users have already tried when they reach the troubleshooting guide: [they have read the basics / they got a specific error code / they tried the recommended steps and it did not work] For each issue: (1) Problem statement (exactly what the user sees — the error message, the behavior, the symptom), (2) Likely causes (in order of frequency — most common first), (3) Diagnostic steps (how to narrow down which cause it is), (4) Fix for each cause (specific, actionable), (5) When to contact support (what to send them — logs, screenshots, environment info). Troubleshooting guides that start with "what you are seeing" instead of "what the system does" get used more.
Pro tip: The first line of every troubleshooting entry should be the exact error message or symptom the user sees — word for word. That is what they searched for. Match their language, not yours.

5. Tutorial Writing

Prompt 10 — Tutorial Outline and Introduction

Help me write a tutorial for a specific task. Task: [describe what the reader will accomplish by the end — e.g., "Connect your app to our API and handle webhook events"] Reader level: [beginner / intermediate — describe their assumed knowledge] Time to complete: [estimated] End state: [exactly what they will have built or achieved] Common mistakes at each step: [list what goes wrong and where] Write: (1) Tutorial introduction (3 sentences — what they will build, what they will learn, what they need to start), (2) Tutorial outline (steps, section titles, estimated time per section), (3) Step 1 in full — with the right level of detail for this audience, (4) "Did it work?" check after each major step (how to verify progress before moving on), (5) Final success state (how to confirm everything is working). Tutorials are about doing, not understanding — every paragraph should move them toward the end state.

Prompt 11 — README File

Write a README file for an open source project or internal tool. Project: [describe what it does in plain language — no proprietary detail] Primary audience: [developers / data engineers / DevOps / general technical users] What they need to do to get started: [describe the install and first-run process at a high level] Key features: [list 4–6] Current stage: [production-ready / beta / experimental / internal tool] README structure: (1) Project title + one-sentence description, (2) Badges section (build status / version / license — describe what to include), (3) What it does (2–3 sentences, plain language), (4) When to use this vs. alternatives, (5) Prerequisites, (6) Installation (step by step), (7) Quick start (get to working state as fast as possible), (8) Features overview, (9) Configuration, (10) Contributing guidelines, (11) License. Also: the most important thing a README can do in its first 10 seconds of reading — and how to achieve it.

6. Career & Process

Prompt 12 — Portfolio Writing Sample

Help me write a documentation sample for my technical writing portfolio. Sample type: [API reference page / getting started guide / conceptual explanation / tutorial / release notes / troubleshooting guide] Topic: [choose a publicly known technology or a fictional product — no proprietary content] Target audience: [describe] What I want this sample to demonstrate: [structure / clarity / tone / audience awareness / complexity I can handle] Write: (1) The complete documentation sample (using a publicly known or fictional product to avoid proprietary concerns), (2) A short "about this sample" paragraph for my portfolio explaining the writing decisions I made, (3) What to do if a hiring manager asks me to explain my process for creating this, (4) How to present this sample in a portfolio (PDF / Confluence demo / GitHub / docs site — which format makes the best impression for different audiences).

Prompt 13 — Technical Writing Job Application

Help me apply for a technical writing role. Job posting: [paste the job description or describe it] My experience: [years / types of docs / tools / industries / products] My strongest samples: [describe what I have] Gaps between the role and my experience: [be honest] Write: (1) Cover letter (under 300 words — leads with the most relevant experience for this specific role, not generic enthusiasm), (2) How to present non-traditional experience as relevant (e.g., writing code comments / developer blog posts / contributing to open source docs), (3) The 5 questions they are most likely to ask in the interview and how to structure strong answers, (4) What to include in the portfolio for this specific role, (5) The single most important thing I should demonstrate in this application given the job description.
Pro tip: Technical writing portfolios that show your process — not just the output — win more interviews. "Here is the doc I wrote" is good. "Here is the doc, here is what the original source material looked like, and here is why I made these choices" is great.

7. Professional Development

Prompt 14 — Learning Plan for a New Tool or Domain

Build a learning plan to expand into a new documentation domain or tool. What I want to learn: [API documentation / docs-as-code / a specific authoring tool / developer documentation / technical illustration / DITA / video scripting / UX writing] My current level: [no experience / some exposure / intermediate] Time available: [hours per week] Goal: [use it professionally within 3 months / add it to my resume / pass a certification / get a job in this specialty] 30-60 day plan: (1) What to learn first vs. what to defer, (2) Best free resources for each learning objective, (3) A practice project to build while learning (something I can add to my portfolio), (4) How to learn the tool in the context of a real documentation workflow (not just the tool features in isolation), (5) How to demonstrate this skill to a hiring manager who will be skeptical of self-taught credentials.

Prompt 15 — Annual Career Review

Help me review my technical writing career this year. Projects completed: [list 4–6] Most impactful documentation I created: [describe — what changed for users or the team because of it] Tools and skills I developed: [list] Collaboration wins and friction: [what worked well, what was hard] What I want to do more of: [type of content / domain / level of technical complexity] Career goal for next year: [senior role / specialist domain / management / freelance / other] Write: (1) Self-review narrative for performance review (400 words — leads with user or business impact, not word count or page count), (2) How to quantify the impact of documentation work (support ticket reduction / onboarding time / error rate — even if you do not have exact numbers, here is how to estimate), (3) How to make the case for what I want next year, (4) The skill or specialty investment that would most differentiate me in the job market, (5) The honest answer: what am I avoiding in my writing that is limiting the quality of my work?
