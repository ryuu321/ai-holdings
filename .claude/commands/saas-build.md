# /saas-build — SaaS開発フレームワーク

SaaSプロダクトを企画から運用まで、品質を落とさずに遂行するための標準プロセス。
引数なしで呼ぶと現在プロジェクトの進捗チェック。引数にプロジェクト名で新規起動。

---

## フェーズ一覧

```
Phase 0: 企画草案
Phase 1: 要件定義
Phase 2: 技術設計
Phase 3: テスト設計（実装前）
Phase 4: 実装
Phase 5: テスト実行・修正
Phase 6: エラーハンドリング
Phase 7: CI設定
Phase 8: デプロイ（ステージング）
Phase 9: 受け入れテスト（UAT）
Phase 10: 本番デプロイ
Phase 11: 運用・モニタリング
```

---

## Phase 0: 企画草案

**目的**: 作る価値があるか判断する。コードは1行も書かない。

確認する項目:
- [ ] **誰が払うか** — 具体的な職種・業種（「中小企業」はNG、「工務店の見積担当者」がOK）
- [ ] **何に困っているか** — 現在の解決手段と、その手段の具体的な不満
- [ ] **なぜAIで解決できるか** — 既存SaaSではなくAIが必要な理由
- [ ] **月いくら払うか** — 代替手段のコスト（時間×時給）と比較して適正単価を出す
- [ ] **¥5M到達の数字** — 単価 × 必要顧客数。現実的か？
- [ ] **競合の薄さ** — 今から参入して勝てる根拠

**GO基準**: 上記すべてに具体的な答えが出たらPhase 1へ。
**STOP基準**: 「誰が払うか」「月いくら」が曖昧なら企画を再考する。

---

## Phase 1: 要件定義

**目的**: 「完成」の定義を決める。後から変えない。

成果物:
- **ユーザーストーリー**: 「〇〇として、△△したい。なぜなら××だから。」の形式で3〜5本
- **受け入れ基準**: 各ストーリーに対して「これができればOK」という具体的条件
- **スコープ外リスト**: 今回作らないものを明示する（機能追加の誘惑を断ち切る）
- **制約条件**: 予算・技術スタック・納期・無料縛り など

**完了基準**: 受け入れ基準を見れば「テストが書けるか」判断できる状態。

---

## Phase 2: 技術設計

**目的**: どう作るかを決める。過剰設計しない。

決めること:
- [ ] **技術スタック** — 既存プロジェクトと揃える（新技術の導入は理由を明記）
- [ ] **ファイル構成** — `app.py` / `{domain}.py` / `tests/` の最小構成
- [ ] **外部依存** — APIキー管理方法・.envと.gitignoreの設計
- [ ] **データフロー** — 入力→処理→出力を1枚の図（テキストでOK）
- [ ] **エラーパス** — 正常系だけでなく、何が壊れ得るかをリストアップ

**原則**: 3行で書けるなら3行で書く。抽象化は3回同じコードを書いてから。

---

## Phase 3: テスト設計（実装前）

**目的**: 何をテストするか先に決める。テストが仕様書になる。

テストの種類と優先度:
1. **単体テスト（必須）** — 純粋関数・バリデーション・変換ロジック
2. **エラーハンドリングテスト（必須）** — API障害・不正入力・クォータ超過
3. **統合テスト（推奨）** — 外部APIはモックせず実際に叩く（ただしCI用のフラグで制御）
4. **E2Eテスト（任意）** — UIがあるなら最低1本のハッピーパス

**禁止事項**:
- テストのためだけに内部実装を変える（テストが実装に依存するのはNG）
- モックだらけのテスト（実際に動かないのに「パス」するテストは無価値）

**完了基準**: テストファイルの骨格（クラス名・テスト名）が書けた状態。

---

## Phase 4: 実装

**目的**: テスト設計に従ってコードを書く。

守るルール:
- [ ] **既存コードを読んでから書く** — 重複・競合の防止
- [ ] **APIキーはハードコードしない** — 必ず環境変数
- [ ] **コメントは「なぜ」だけ** — 「何をしているか」は変数名で伝える
- [ ] **1関数1責任** — 生成・検証・整形は別関数に分ける
- [ ] **エラーパスを実装に含める** — Phase 2で洗い出した障害ケースを必ず実装

**完了基準**: `python app.py` または `streamlit run app.py` でローカル動作する。

---

## Phase 5: テスト実行・修正

**目的**: Phase 3で設計したテストをすべてパスさせる。

手順:
1. `pytest tests/ -v` を実行
2. 失敗したテストは実装を修正する（テストを緩めない）
3. カバレッジが薄い箇所があれば追加テストを書く
4. 全テストパスを確認してからコミット

**完了基準**: `pytest tests/ -v` で全テストグリーン。

---

## Phase 6: エラーハンドリング

**目的**: 本番で壊れたときユーザーに意味のあるメッセージを返す。

チェックリスト:
- [ ] **外部API障害** — 429/500/タイムアウトを個別に捕捉して `ok: False` で返す
- [ ] **クォータ超過** — リトライせず即座に返す（無駄なリトライは問題を悪化させる）
- [ ] **ネットワーク断** — リトライ回数上限を設ける（無限ループ禁止）
- [ ] **不正入力** — ユーザー入力は信頼しない。境界値を確認する
- [ ] **エラーメッセージ** — 技術的な例外をそのまま見せない。ユーザーが次にやるべきことを示す

**完了基準**: エラーハンドリングのテストがすべてパスする。

---

## Phase 7: CI設定

**目的**: pushのたびに自動でテストが走る状態にする。

`.github/workflows/{project}-test.yml` の最小構成:
- トリガー: `push` / `pull_request` (対象パスのみ)
- Python環境のセットアップ
- `pip install -r requirements.txt pytest`
- `pytest tests/ -v --tb=short`
- APIキーはダミー値を `env:` に設定（テストはモック済みのため実APIは不要）

**完了基準**: GitHubのActionsタブでグリーンになる。

---

## Phase 8: デプロイ（ステージング）

**目的**: 実際のURLで動作を確認する。本番ではない。

Streamlit Cloudの場合:
1. `streamlit.io/cloud` でリポジトリを接続
2. `secrets.toml` にAPIキーを設定（GitHubには入れない）
3. アプリのURLを取得
4. ハッピーパスを1件手動で動かす

**完了基準**: 公開URLで実際に生成が動く。

---

## Phase 9: 受け入れテスト（UAT）

**目的**: Phase 1の受け入れ基準を実際に満たしているか確認する。

手順:
1. Phase 1の受け入れ基準リストを開く
2. 1件ずつ実際に操作して確認する
3. NGがあれば実装に戻る（「だいたい動く」は完了ではない）

**完了基準**: 受け入れ基準の全項目が実際の操作で確認できた。

---

## Phase 10: 本番デプロイ

**目的**: 実ユーザーに届ける状態にする。

チェックリスト:
- [ ] ステージングと本番で同じ挙動
- [ ] エラー時のメッセージが非技術者にも伝わる
- [ ] 料金・プラン・トライアル期間が明示されている（無料トライアルの範囲を決める）
- [ ] 問い合わせ先（メール等）が表示されている

---

## Phase 11: 運用・モニタリング

**目的**: 壊れたときに誰が・何を見て気づくか決める。

最低限やること:
- [ ] **エラーログ** — 生成失敗・APIエラーを記録する仕組み
- [ ] **週次確認** — APIクォータ残量・課金状況を確認する（週10分）
- [ ] **フィードバック導線** — ユーザーが不満を伝えられる場所（メール/フォーム）
- [ ] **解約防止** — 30日使っていないユーザーへのリマインド（将来的に）

---

## 絶対ルール

- **フェーズを飛ばさない** — Phase 3（テスト設計）をスキップした実装は受け入れない
- **「だいたい動く」は完了ではない** — 受け入れ基準を満たすまで次に進まない
- **テストを緩めてパスさせない** — 実装を直す。テストの期待値を下げない
- **コードより先に仕様を変えない** — 途中で要件が変わったらPhase 1に戻る
